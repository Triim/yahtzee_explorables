# Yahtzee Explorable — Архитектура и техническая часть

Опирается на `yahtzee-explorable-skeleton.md` и `yahtzee-explorable-spec.md`. Этот файл — для непосредственной реализации в Claude Code.

---

## 1. Принципиальные решения (зафиксировано)

- **Стек:** Vite + React + TypeScript. Стили — минимальный CSS (CSS-переменные для токенов); тяжёлые UI-библиотеки не нужны.
- **Формулы:** KaTeX (рендер LaTeX из спеки).
- **Скролл:** нативный, `position: sticky` для правой панели. **Никакого scroll-hijacking.** Смена сцен — через IntersectionObserver на текстовых шагах.
- **Графика:** SVG, рисуем **вручную**, единым визуальным языком. Никаких chart-библиотек (recharts/chart.js) — их стиль будет выбиваться и его придётся переопределять.
- **Тяжёлый DP:** считаем офлайн на этапе сборки → отгружаем как оракул (таблица значений). В рантайме — только просмотр.
- **Монте-Карло:** в web-worker (тысячи партий), чтобы не вешать скролл.
- **Адаптивность:** desktop-first. Мобилка — заглушка «откройте на компьютере», без полноценной вёрстки.
- **Тон:** чистая минималистичная вёрстка (см. §7).

---

## 2. Разделение слоёв

Жёстко разделяем **движок** (чистая математика, без React) и **презентацию** (React + SVG).

```
src/
  engine/            // чистый TS, без зависимостей от React
    outcomes.ts      // пространство исходов, частоты
    distributions.ts // распределения, биномиальное, нормаль-оверлей
    combinatorics.ts // мультимножества (252), наборы кип-ов (462)
    conditional.ts   // условные распределения переброса
    scoring.ts       // категории как предикаты, score = f(hand)
    ev.ts            // мат. ожидание
    oracle.ts        // загрузка/просмотр предпосчитанной V-таблицы
    montecarlo.ts    // прогон партий (используется и в воркере)
    types.ts         // Hand, State, Category, Roll, Policy, ...
  worker/
    montecarlo.worker.ts
  scenes/            // по сцене: модель (правая панель) + конфиг шагов
  components/        // переиспользуемые виджеты из словаря компонентов
  scaffolding/       // Scrollytelling-каркас (Scene, Step, observer)
  state/             // глобальный стор (IntuitionNotebook)
  styles/            // токены, layout
  data/              // оракул (JSON/бинарь), предпосчитанные таблицы
```

Правило: движок ничего не знает про React и DOM; его можно гонять в воркере и в build-скрипте. Компоненты — тонкие обёртки над движком.

Движок проектируется **сразу под финальный солвер**, а не наращивается от сцены к сцене: `Die` из Scene 1 не должен диктовать структуру `State` из Scene 7. Презентация наращивается по фазам, движок — цельный с начала.

## 3. Scrollytelling: контракт Scene / Step

Левая колонка скроллится, правая залипает и меняет модель на порогах. Реализация — `IntersectionObserver` на шагах.

```ts
type Register = 'free' | 'driven' | 'showcase';

type Directive =
  | { kind: 'activate'; model: string }
  | { kind: 'unlock'; tool: string }
  | { kind: 'setState'; payload: unknown }
  | { kind: 'showcase'; clip: string };

interface Step {
  id: string;            // 'S1.4'
  copyType: 'вопрос' | 'инструкция' | 'определение' | 'формула' | 'врезка' | 'переход';
  register: Register;
  directive?: Directive; // что сделать с правой панелью при входе шага во вьюпорт
  // текст шага — из спеки (рендер с KaTeX для формул)
}

interface Scene {
  id: string;            // 'scene-1'
  model: React.FC<SceneModelProps>; // правая панель
  steps: Step[];
  panel: 'single' | 'rich'; // rich = модель + связанный график в одной правой колонке
}
```

**Механика:**
- Каждый `Step` — DOM-элемент с `data-step-id`. Один `IntersectionObserver` (rootMargin ~ центр вьюпорта) определяет активный шаг.
- При смене активного шага исполняется его `directive`: сменить модель (`activate`), добавить инструмент (`unlock`), задать состояние (`setState`) или запустить демонстрацию (`showcase`).
- **driven**-шаги: параметр модели привязан к прогрессу скролла внутри шага (доля видимости), а не к кликам.
- **Сцены перетекают:** последний шаг сцены — это `[переход]`-вопрос; на следующий скролл меняется модель (правая панель), а текст продолжается. Контент непрерывен.

**«Богатая правая панель» вместо трёх колонок:** при `panel: 'rich'` правая колонка вертикально содержит модель + связанный график (например, `WinRegionChart` под `OpponentPanel`). Лэйаут остаётся двухколоночным — это спокойнее и лучше переживает узкие экраны.

## 4. Регистры взаимодействия (инвариант честности)

- **free** — модель интерактивна, результат честный. Текст шага не должен утверждать ничего, что зависит от состояния, которое пользователь мог изменить.
- **driven** — модель управляется скроллом; интерактив пользователя отключён или ограничен.
- **showcase** — постановочная анимация. **Обязательно** визуально отделена (рамка/подпись «демонстрация») и не имеет кнопки «бросить», имитирующей честный результат. Это техническое требование, а не косметика: showcase-клип проигрывается из заранее заданного сценария и явно помечен.

Тип `Register` — на уровне `Step`; компонент-обёртка сцены читает его и переключает поведение модели и маркировку.

## 5. Математический движок и DP-оракул

**Состояния (точные числа для движка):**
- Различимых бросков: `C(6+5-1,5) = 252` (мультимножества).
- Наборов кип-ов: `C(7+5-1,5) = 462`.
- Состояний внутри хода: `3·252 + 2·462 = 1680`.
- Счётных состояний партии: `3·2^12·64 = 786 432`, ужимается до `310 656`.
- Оптимальное ожидание под одиночной стратегией ≈ `254.589`.

**Оракул (предпосчёт офлайн):**
- Build-скрипт (Node, использует тот же `engine/`) считает обратной индукцией `V` для счётных состояний и сохраняет в `data/` (компактный бинарь или JSON).
- В рантайме: загрузить таблицу один раз; значение **состояния внутри хода** быстро пересчитывается из значений счётных состояний (по формулам spec-движка), его не храним.
- Для оптимизации перебора кип-ов использовать рекуррентность:
  `keep[K] = roll[K] (|K|=5)` либо `(1/6)·Σ_d keep[K∪{d}]` (|K|<5);
  `roll[K] = max({keep[K]} ∪ {roll[K\{d}] | d∈K})`.
- Если полная таблица велика для отгрузки в браузер — допустимо отгружать **редуцированный оракул** (только состояния/политики, нужные демонстрационным сценам 7–10), а не весь граф. Решить на этапе сборки по фактическому размеру.

**Бонусы в движке:** верхний `+35` при сумме верхней секции `≥63`; Yahtzee/Joker `+100` за каждый дополнительный Yahtzee при заполненной Yahtzee-ячейке (50). Учитывать в `scoring.ts` и в состоянии (тернарный флаг + сумма верхней секции `[0,63]`).

## 6. Монте-Карло (web-worker)

- `montecarlo.worker.ts` гоняет тысячи партий для `ChampionshipRunner` (Scene 8) и для оценок в Scene 9–10.
- Воркер импортирует чистый `engine/` (тот же код, что и в оракуле).
- Возвращает гистограммы итогов и подписи `(μ, σ)` ролей; UI рисует их прогрессивно (не блокируя).
- Это и есть callback к ЗБЧ: среднее по симуляциям сходится — `P̂ = (1/N)Σ 1[·]`.
- Реальное распределение итогов имеет тяжёлый правый хвост от Yahtzee-бонусов — `ScoreHistogram` показывает и `NormalOverlay` (идеализация массы), и `TailMarker` (реальный хвост, вплоть до showcase-максимума 1575).

## 7. Минималистичная вёрстка (визуальная директива)

- **Два столбца:** текст (лево, скроллится) | модель (право, sticky). Щедрые поля и воздух; один смысловой блок на экран.
- **Типографика:** один чистый шрифтовой стиль, крупный читаемый размер для текста, моноширинный/математический для формул (KaTeX). Минимум жирного.
- **Цвет:** нейтральный фон, один акцентный цвет. В диаграммах допускается семантическая палитра (холодный/горячий узлы, узкая/широкая стратегия), но скупо.
- **SVG-стиль единый:** кубики, гистограммы, граф состояний, воронка — один язык линий и заливок. Тонкие штрихи, скруглённые узлы, спокойные заливки.
- **Движение сдержанное:** анимации только там, где они объясняют (бросок, схлопывание воронки, обратная индукция). Никаких декоративных эффектов.
- **Тёмная тема:** через CSS-переменные (фон/текст/границы), цвета кривых — через мид-тона, работающие в обеих темах.

## 8. Состояние приложения

- **Локально по сцене:** состояние модели (текущие кубики, выбор кип-ов, параметр слайдера). Сбрасывается при входе в сцену, если иное не указано.
- **Глобально:** `IntuitionNotebook` (ответы Scene 0) — React Context, in-memory. Нужен для поздних callback-ов («помнишь свою догадку?»).
- **Без localStorage/sessionStorage** в артефактном окружении; если собирается как самостоятельное приложение — персист не требуется по смыслу, можно не вводить.

---

## 9. Производительность

- Монтируется/считается только активная сцена (по IntersectionObserver); неактивные — размонтированы или с приостановленным вычислением. На скролле ничего тяжёлого не пересчитывать.
- Оракул грузится один раз, лениво, перед Scene 7.
- Монте-Карло — только в воркере; UI рисует результаты порциями.
- Мемоизировать чистые вычисления движка (распределения, EV) по входу.

---

## 10. Доступность и честность

- Нативный скролл и `sticky` → корректная работа клавиатуры/скринридеров; никакого перехвата скролла.
- Каждый SVG — с `role="img"`, `title`/`desc`.
- Интерактивные элементы (кнопки броска, тумблеры) — настоящие `<button>`/контролы.
- `showcase`-режим помечен и текстом, и визуально; честный `free`-бросок никогда не подменяется сценарием.

---

## 11. Порядок реализации (фазы)

- **Фаза 1 — Scenes 0–5.** Каркас scrollytelling (`Scene`/`Step`/observer), движок (исходы, распределения, мультимножества, условное распределение, EV), презентационные компоненты. Выпускаемый минимум.
- **Фаза 2 — Scenes 6–7.** Полный ход, бонусы, build-скрипт оракула, `StateGraph` с холодными/горячими узлами, `V`-просмотр.
- **Фаза 3 — Scenes 8–10.** Монте-Карло в воркере, чемпионат, нормаль+хвост, showcase 1575, противник, win-region, воронка, адаптивный финал, капстоун.

Граница Фаз 1/2 — естественный шов (одиночные идеи → партия целиком). Движок строится сразу под Фазу 3.

---

## 12. Что НЕ делать (тех. инварианты)

- Не вводить scroll-hijacking и кастомную инерцию скролла.
- Не тащить chart-библиотеки.
- Не имитировать честный результат в showcase.
- Не пересчитывать DP в браузере на лету — только просмотр предпосчитанного оракула.
- Не делать полноценную мобильную вёрстку — только заглушка.
- Не хранить состояние в localStorage в артефактном окружении.
